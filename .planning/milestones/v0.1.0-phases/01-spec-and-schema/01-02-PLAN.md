**spec/vectors/query.json** — operation "query":

test_groups:
1. "happy path" tests:
   - tc_id 1: insert 2 events with known timestamps (as quoted strings like "1710000000000000001"), query range that includes both, expects returned_count 2 with events in insertion order. Events array in expected includes ts and payload for each.
   - tc_id 2: insert 3 events, query range that includes only the middle one, expects returned_count 1
   - tc_id 3: query with start == end matching exactly one event's timestamp, expects returned_count 1 (verifies inclusive range)

2. "empty results" tests:
   - tc_id 4: insert events for topic "sensor", query topic "other" (no events), expects returned_count 0, events []
   - tc_id 5: insert event at ts "1710000000000000001", query range "2000000000000000000" to "2000000000000000999" (no overlap), expects returned_count 0, events []

3. "error cases" tests:
   - tc_id 6: store not open, expects error_kind "store_not_open"

ALL timestamps MUST be quoted strings (e.g., "1710000000000000001"), NEVER JSON numbers. This is critical for JavaScript 53-bit precision safety (SPEC-03).

ALL payloads MUST be RFC 4648 standard base64.

**spec/vectors/immutability.json** — operation "immutability":

This file tests storage-level enforcement. The harness must execute raw SQL (bypassing SDK public API).

test_groups:

   - tc_id 2: setup inserts one event, attempts "DELETE FROM events WHERE id = 1", expects error_kind "immutability_violation" with error_message_contains "cairn: deletes not allowed"

The setup field is an array of {topic, payload} objects. The sql field contains the raw SQL to execute. The expected result is "invalid" with the trigger error message.
